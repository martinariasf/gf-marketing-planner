2. **Misleading `slide_index` in return on skip-ahead** (`_link_slide_to_post`). When `slide_index` exceeds the current deck length (e.g., caller requests index 3 with only 1 slide), the slide is appended at the end (position 2), but the return value still reports `"slide_index": 3` (the requested index, not the actual position). The agent could believe the slide landed at position 3. The `slide_count` from the confirm GET is accurate, but the mismatch is still confusing.

3. **`linked` confirmation does not verify slide position** (`_link_slide_to_post`). The GET confirmation checks `any(s.get("image") == url for s in served_slides)` — it confirms the URL exists *somewhere* in the deck, not at the requested index. A slide placed at the wrong position (due to skip-ahead) would still report `linked: True`, masking the misplacement.

4. **Single-slide carousel sets `format: "carousel"` unconditionally** (`_link_slide_to_post`). After the first `image_generate(post_id, slide_index=1)` call, the post is patched with `format: "carousel"` and a 1-element `slides[]`. While this satisfies criterion 7 (non-empty slides after interruption), a 1-slide "carousel" is technically invalid per the config's "2–10 images" rule. The acceptance criteria don't require a minimum, but if the agent is interrupted after slide 1, the post is left as a 1-slide carousel — arguably a degraded state worth noting (though the config's preview-step workflow acknowledges this).

5. **PATCH body sends `format: "carousel"` explicitly** (`_link_slide_to_post`). Acceptance criterion 8 says "coalescePost derives format" — the API should infer carousel from `slides[]`. Sending `format` explicitly is redundant. If the API's `postPatchSchema` rejects `format` as a field (accepting only `slides`), this would 422. The old config showed the agent PATCHing with `format`, so the API likely accepts it, but this should be confirmed against the schema.
